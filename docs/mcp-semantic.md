# MCP Semantic Server — `dbt_core_semantic_layer`

**Package:** `src/mcp_semantic_healh_gen_ai_chat`  
**Transport:** stdio  
**FastMCP name:** `dbt_core_semantic_layer`

## Overview

The Semantic MCP server acts as the **discovery layer** for the entire system. It reads the two artifact files produced by `dbt compile` / `dbt parse` and exposes them to the LLM as structured tools. The LLM calls these tools first — before writing any SQL — to understand what data exists, how it is structured, and which columns are available.

```
dbt project
  └── dbt compile
        ├── target/manifest.json          ← model & column metadata
        └── target/semantic_manifest.json ← metrics & dimensions
              │
              └── mcp_semantic (reads both)
                    │
                    └── LLM tools
```

## Configuration

| Environment variable | Default | Description |
|---|---|---|
| `DBT_MANIFEST_PATH` | `../../dbt_health_gen_ai_chat/target/manifest.json` | Path to `manifest.json` generated by `dbt compile` |
| `DBT_SEMANTIC_MANIFEST_PATH` | `../../dbt_health_gen_ai_chat/target/semantic_manifest.json` | Path to `semantic_manifest.json` generated by `dbt compile` |

Both paths are resolved relative to the working directory when the server process starts. Override them with absolute paths to avoid ambiguity.

## Starting the server

```bash
# Standalone (from repo root — for testing with an MCP inspector)
uv run python -m src.mcp_semantic_healh_gen_ai_chat.main

# With custom artifact paths
DBT_MANIFEST_PATH=/data/dbt/target/manifest.json \
DBT_SEMANTIC_MANIFEST_PATH=/data/dbt/target/semantic_manifest.json \
uv run python -m src.mcp_semantic_healh_gen_ai_chat.main
```

> The server is normally started automatically by the chat agent — you do not need to run it manually.

## Tools

### `list_local_metrics()`

Lists every metric defined in `semantic_manifest.json`.

**Parameters:** none

**Returns:** JSON array of metric objects.

```json
[
  {
    "name": "glucose_value_average",
    "description": "The average glucose value",
    "type": "simple",
    "expression": "glucose_value_average"
  }
]
```

**Typical LLM use:** Called at the start of a conversation to understand what business-level measures are defined before deciding how to query the data.

---

### `get_dimensions_by_semantic_model()`

Returns all dimensions and their types for every semantic model in `semantic_manifest.json`.

**Parameters:** none

**Returns:** JSON object keyed by semantic model name.

```json
{
  "semantic_glucose_register": [
    { "name": "timestamp_day", "type": "time" },
    { "name": "timestamp_hour", "type": "categorical" },
    { "name": "sensor_scan", "type": "categorical" }
  ],
  "semantic_strava_activities": [
    { "name": "start_date_local", "type": "time" }
  ]
}
```

**Typical LLM use:** Used to understand which fields can be used to slice and filter data in queries.

---

### `get_model_lineage(model_name: str)`

Returns metadata and upstream dependencies for a named dbt model by reading `manifest.json`.

**Parameters:**

| Parameter | Type | Description |
|---|---|---|
| `model_name` | `str` | The dbt model name (e.g. `"view_glucose_register"`) |

**Returns:** JSON object with the model's database, schema, and list of upstream node IDs.

```json
{
  "name": "view_glucose_register",
  "database": "health_db",
  "schema": "dbt_dev",
  "upstream_dependencies": [
    "source.health_gen_ai_chat.abbot.glucose_register"
  ]
}
```

**Returns** `"Model '<name>' not found in manifest.json."` if the model does not exist.

**Typical LLM use:** Used to trace where a model's data comes from before writing queries against it.

---

### `get_table_columns(model_name: str)`

Returns column names and data types for a named dbt model or source. Searches `nodes` first, then `sources` in `manifest.json`.

**Parameters:**

| Parameter | Type | Description |
|---|---|---|
| `model_name` | `str` | The dbt model name (e.g. `"view_glucose_register"`) or source name (e.g. `"glucose_register"`) |

**Returns:** JSON object with database/schema location and a list of columns.

```json
{
  "model": "view_glucose_register",
  "database": "health_db",
  "schema": "dbt_dev",
  "columns": [
    { "name": "uuid", "data_type": "string", "description": "The unique identifier for each record" },
    { "name": "timestamp", "data_type": "datetime", "description": "The timestamp of the glucose measurement" },
    { "name": "glucose_value", "data_type": "double", "description": "The glucose level measurement" },
    { "name": "sensor_scan", "data_type": "tinyint", "description": "Sensor scan flag (1 = scan, 0 = manual)" },
    { "name": "timehour", "data_type": "unknown", "description": "Hour of day (0-23), derived by dbt — type not tracked in manifest" },
    { "name": "timestamp_day", "data_type": "unknown", "description": "Date only, derived by dbt — type not tracked in manifest" }
  ]
}
```

**Returns** an error string if the model or source is not found.

**Typical LLM use:** Called immediately before writing a SQL query to confirm exact column names, avoiding hallucinated field names.

## Data sources exposed

The dbt project currently defines two sources:

### `abbot.glucose_register` (schema: `cgm_abbot_connector`)

Continuous glucose monitor readings from the Abbott LibreLink connector.

| Column | Type | Description |
|---|---|---|
| `uuid` | string | Unique record identifier |
| `row_created_at` | datetime | Row insertion timestamp |
| `timestamp` | datetime | When the glucose was measured |
| `glucose_value` | double | Glucose level (mg/dL or mmol/L) |
| `sensor_scan` | tinyint | `1` = sensor scan, `0` = manual reading |

dbt view: `view_glucose_register` — adds derived columns `timehour` (hour of day) and `timestamp_day` (date only).

### `strava.strava_activities` (schema: `strava-to-db`)

Physical activity records synced from Strava.

| Column | Type | Description |
|---|---|---|
| `activity_id` | int | Unique activity identifier |
| `name` | string | Activity title |
| `activity_type` | string | General type (Run, Ride, Swim, …) |
| `sport_type` | string | Detailed sport classification |
| `start_date` | datetime | UTC start time |
| `start_date_local` | datetime | Local start time |
| `distance_m` | double | Distance in metres |
| `moving_time_s` | int | Moving time in seconds |
| `elapsed_time_s` | int | Total elapsed time in seconds |
| `total_elevation_gain_m` | double | Elevation gain in metres |

dbt view: `view_strava_activities` — passes all columns through unchanged.

## Semantic models

| Semantic model | Source view | Primary entity | Measures | Dimensions |
|---|---|---|---|---|
| `semantic_glucose_register` | `view_glucose_register` | `uuid` | `glucose_value_average` (avg) | `timestamp_day` (time), `timestamp_hour` (cat), `sensor_scan` (cat) |
| `semantic_strava_activities` | `view_strava_activities` | `activity_id` | `distance_m_sum` (sum) | `start_date_local` (time) |

## Error handling

All tools catch all exceptions and return a plain error string rather than raising. The LLM should detect error strings (they begin with `"Error reading..."` or `"Model '...' not found"`) and report them to the user.

The most common error is a missing artifact file — ensure `dbt compile` has been run before starting the chat agent.
