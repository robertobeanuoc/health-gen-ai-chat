# MCP Exec Server — `mysql_execution_engine`

**Package:** `src/mcp_exec_health_gen_ai_chat`  
**Transport:** stdio  
**FastMCP name:** `mysql_execution_engine`

## Overview

The Exec MCP server is the **query execution layer**. It receives SQL `SELECT` statements generated by the LLM, runs them against a MySQL database via SQLAlchemy, and returns the results as a JSON array. Write and structural operations are blocked at the application level.

```
LLM
  └── execute_read_query(sql)
        └── SQLAlchemy (PyMySQL driver)
              └── MySQL database
```

## Configuration

| Environment variable | Required | Description |
|---|---|---|
| `MYSQL_ALCHEMY_URI` | **Yes** | SQLAlchemy connection URL for the MySQL database |

### Connection URL format

```
mysql+pymysql://<user>:<password>@<host>:<port>/<database>
```

Examples:

```bash
# Local database
MYSQL_ALCHEMY_URI="mysql+pymysql://root:secret@localhost:3306/health_db"

# Remote database
MYSQL_ALCHEMY_URI="mysql+pymysql://analyst:pass@db.example.com:3306/health_db"

# With SSL (append query params)
MYSQL_ALCHEMY_URI="mysql+pymysql://user:pass@host/db?ssl_ca=/certs/ca.pem"
```

> The engine is **lazy-initialized**: the server process starts successfully even if `MYSQL_ALCHEMY_URI` is not set. The error is raised only when the first query is attempted. This allows the chat agent to start all three MCP servers without needing the database credentials available at process start time.

## Starting the server

```bash
# Standalone (from repo root)
MYSQL_ALCHEMY_URI="mysql+pymysql://user:pass@localhost/health_db" \
  uv run python -m src.mcp_exec_health_gen_ai_chat.main

# Test connection manually
uv run python -c "
from sqlalchemy import create_engine, text
import os
e = create_engine(os.environ['MYSQL_ALCHEMY_URI'])
with e.connect() as c:
    print(c.execute(text('SELECT 1')).fetchone())
"
```

> The server is normally started automatically by the chat agent.

## Tools

### `execute_read_query(sql_query: str)`

Executes a read-only SQL statement and returns all rows as a JSON array.

**Parameters:**

| Parameter | Type | Description |
|---|---|---|
| `sql_query` | `str` | A valid MySQL `SELECT` statement |

**Returns:** A JSON string — either a list of row objects or an error string.

**Success response:**

```json
[
  { "timestamp_day": "2024-03-01", "avg_glucose": 105.3 },
  { "timestamp_day": "2024-03-02", "avg_glucose": 98.7 }
]
```

**Error responses:**

```
Security error: Query contains forbidden keywords. Only SELECT processing is allowed.
```

```
Execution error in MySQL database: (pymysql.err.OperationalError) ...
```

**Typical LLM use:**

```sql
SELECT DATE(timestamp) AS day, AVG(glucose_value) AS avg_glucose
FROM cgm_abbot_connector.glucose_register
WHERE timestamp >= DATE_SUB(NOW(), INTERVAL 7 DAY)
GROUP BY day
ORDER BY day;
```

## Security

The server enforces a **keyword blocklist** before executing any query. The following keywords cause an immediate rejection without touching the database:

| Blocked keyword | Why |
|---|---|
| `INSERT` | Prevents data insertion |
| `UPDATE` | Prevents record modification |
| `DELETE` | Prevents record deletion |
| `DROP` | Prevents table/database deletion |
| `ALTER` | Prevents schema changes |
| `GRANT` | Prevents privilege escalation |
| `REPLACE` | Prevents insert-or-replace operations |
| `TRUNCATE` | Prevents table clearing |

The check is case-insensitive and applied to the entire query string before any parsing.

> **Note:** The blocklist is a first line of defense suitable for trusted LLM usage. For production deployments with untrusted inputs, pair this with a MySQL user that has only `SELECT` privileges on the target schemas.

## Connection pooling

SQLAlchemy's default connection pool is used with `pool_pre_ping=True`. This means:

- A lightweight `SELECT 1` is issued before each query to validate the connection is still alive.
- Stale connections (e.g. after a MySQL server restart or idle timeout) are transparently recycled.
- No manual keep-alive logic is needed.
